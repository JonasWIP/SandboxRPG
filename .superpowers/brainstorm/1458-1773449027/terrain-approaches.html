Terrain & Chunk System — 3 Approaches

Pick the architecture that feels right. My recommendation is C.

A

Pre-generate everything on Init()

Fixed world size (e.g. 1600×1600m = 50×50 chunks). Server generates all chunks at startup and stores them in SpacetimeDB. Clients subscribe and receive chunks as they explore.

Pros

  • Simple — no on-demand logic
  • Every chunk ready instantly

Cons

  • Slow startup (generating 2500 chunks)
  • All data in RAM from the start
  • Fixed world size forever
B

Lazy on-demand generation

Chunks are generated the first time any player steps into that area. Server stores a "generated" flag per chunk. Resource nodes spawn when the chunk is first created. World can be very large.

Pros

  • Memory efficient — only explored areas live in DB
  • True exploration feel
  • Scales to any world size

Cons

  • First player in area gets a generation spike
  • More complex chunk state machine
C ★ Recommended

Hybrid — pre-generate town core, lazy outer world

On Init(), eagerly generate the spawn town zone (fixed area ~200×200m) so it's always ready. The outer world uses lazy on-demand generation as players explore. The biome noise map is seeded once and stored — so chunk biome type is always deterministic even before generation.

Pros

  • Town is always stable and ready
  • Outer world scales with exploration
  • Fast startup
  • Biomes are consistent across the whole map

Cons

  • Two generation paths to implement (eager + lazy)
  • Slightly more complex than pure A or B

Why C?

The spawn town needs to be deterministic and instantly available when the server starts — you don't want the first player in to trigger a generation spike in the hub. The outer world benefits from lazy loading because most of it will never be visited. Storing the biome seed up front means the whole map is conceptually "known" (you can show a world map), just not yet materialized in the DB.
