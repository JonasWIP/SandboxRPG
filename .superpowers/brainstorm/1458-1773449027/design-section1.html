Design — Section 1: World Overview & Chunk System

Does this structure look right before we go deeper?

World Size & Coordinates

  • Chunk size: 32m × 32m
  • Town zone: ±5 chunks from origin (320×320m) — pre-generated on Init()
  • Outer world: effectively unbounded, lazy-generated up to a soft cap (~200 chunks radius)
  • Y (height): 0–30m range, sea level at 2m

New SpacetimeDB Tables

// One row — world seed & config
WorldConfig { Seed, ChunkSize, TownRadius }

// One row per generated chunk
WorldChunk {
ChunkKey (PK, packed X|Z),
ChunkX, ChunkZ,
BiomeType,
HeightData (byte[], 33×33),
IsTownZone,
IsGenerated
}

// One row per node (tree/rock/ore)
ResourceNode {
Id (AutoInc PK),
ChunkKey,
NodeType, ResourceType,
PosX, PosY, PosZ,
Health, MaxHealth,
RespawnAt (0 = alive)
}

Generation Flow

  1. Server Init(): store WorldConfig with random seed, pre-generate town chunks
  2. Player moves → server checks if chunk exists in DB
  3. If not: generate heightmap + biome + resource nodes, insert into DB
  4. Client subscribes to WorldChunk + ResourceNode → builds terrain mesh + spawns node visuals
  5. Player harvests node → node health decreases → at 0, set RespawnAt timestamp
  6. Scheduled reducer checks RespawnAt, restores node health when timer expires
